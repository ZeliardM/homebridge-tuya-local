# AGENTS.md — AI Coding Agent Guide

This file helps AI coding agents understand the structure, conventions, and workflows of this project.

## Project Overview

**homebridge-tuya-local** is a Homebridge plugin that controls Tuya-based smart devices over LAN without cloud dependency. It communicates directly with devices using their Device ID and Local Key via the Tuya protocol (versions 3.1, 3.3, 3.4).

- **Plugin name**: `homebridge-tuya-local`
- **Platform alias**: `TuyaLocalPlatform` (used in Homebridge `config.json` and `config.schema.json`)
- **Package**: `homebridge-tuya-local`
- **License**: MIT

## Tech Stack

- **Language**: TypeScript (strict mode, `strictNullChecks` and `strictFunctionTypes` off)
- **Module system**: CommonJS (output to `dist/`)
- **Target**: ES2022
- **Runtime**: Node.js >= 18, Homebridge >= 1.6.0
- **Documentation**: VitePress (source in `docs/`, deployed to GitHub Pages)

## Directory Structure

```text
src/                                → TypeScript source (rootDir)
  index.ts                          → Plugin entry point, registers platform with Homebridge
  types.ts                          → Shared type definitions (all interfaces, enums, type aliases)
  declarations.d.ts                 → Ambient module declarations
  protocol/                         → Tuya LAN communication layer
    TuyaAccessory.ts                → TCP protocol client (connect, encrypt, send/receive DPs)
    TuyaDiscovery.ts                → UDP device discovery on local network
  accessories/                      → HomeKit accessory implementations
    index.ts                        → Barrel re-exports for all accessories
    Base.accessory.ts               → Abstract base class all accessories extend
    EnergyCharacteristics.ts        → Shared energy monitoring (volts/amps/watts) factory
    {DeviceType}.accessory.ts       → Device-specific accessory implementations (23 types)
  assets/                           → Static assets (logos)
dist/                               → Compiled JS output (gitignored, generated by `npm run build`)
docs/                               → VitePress documentation source
  .vitepress/config.mts             → VitePress configuration (ESM, .mts extension required)
  index.md                          → Docs home page
  *.md                              → Documentation pages
config.schema.json                  → Homebridge Config UI X schema (defines UI for plugin settings)
```

## Architecture

### Plugin Registration

`src/index.ts` exports a function that Homebridge calls. It:

1. Extracts HAP services/characteristics from the Homebridge API.
2. Calls `homebridge.registerPlatform()` with `PLUGIN_NAME` and `PLATFORM_NAME`.
3. Maps device type strings (case-insensitive) to accessory classes via `CLASS_DEF`.

### Accessory Pattern

Every device type follows the same pattern:

1. Extends `BaseAccessory` (which manages the Homebridge `PlatformAccessory` lifecycle).
2. Implements `_registerPlatformAccessory()` to create HomeKit services and characteristics.
3. Reads/writes Tuya DataPoints (DPs) — numeric keys that map to device functions.
4. Receives DP updates from `TuyaAccessory` and pushes state changes to HomeKit.

### Device Type Registry

The `CLASS_DEF` map in `src/index.ts` maps lowercase type strings to classes:

```text
outlet, simplelight, rgbtwlight, rgbtwoutlet, twlight, multioutlet,
custommultioutlet, airconditioner, airpurifier, dehumidifier, convector,
garagedoor, simpledimmer, simpledimmer2, simpleblinds, simpleheater,
mappedheatpumpheater, circuitbreakermonitor, switch, fan, fanlight,
watervalve, oildiffuser
```

Users set the `type` field in their config (case-insensitive). The platform lowercases it and looks up the class.

### Communication Layer

- **TuyaAccessory** (`protocol/TuyaAccessory.ts`): Handles TCP connections to individual devices, encryption/decryption, and DP state management. Not a HomeKit accessory — it's the protocol client.
- **TuyaDiscovery** (`protocol/TuyaDiscovery.ts`): Listens for UDP broadcasts from Tuya devices on the LAN to discover IPs.

### Energy Monitoring

`accessories/EnergyCharacteristics.ts` is a factory that adds voltage, current, and power characteristics to outlets and outlet-like accessories that support energy readings.

## Conventions

- **Indentation**: 2 spaces everywhere (TS, JS, JSON, YAML, MD).
- **Line endings**: LF.
- **Naming**: PascalCase for classes, camelCase for variables/functions. Accessory files use the `{DeviceType}.accessory.ts` convention. Protocol files keep PascalCase names.
- **No external Tuya libraries**: The project implements the Tuya LAN protocol directly in `protocol/TuyaAccessory.ts` and `protocol/TuyaDiscovery.ts`. It does NOT depend on `tuyapi` or `@tuyapi/cli`.
- **Config schema**: `config.schema.json` at project root defines the Homebridge Config UI X form. Keep it in sync with any new device types or parameters.
- **VitePress config**: Must use `.mts` extension (not `.ts`) because the project is CommonJS but VitePress is ESM-only.

## Adding a New Device Type

1. Create `src/accessories/{DeviceName}.accessory.ts` extending `BaseAccessory`.
2. Implement `_registerPlatformAccessory()` with the appropriate HomeKit services.
3. Export the class as `default` and add a re-export in `src/accessories/index.ts`.
4. Add the lowercase type key → class mapping in `CLASS_DEF` in `src/index.ts`.
5. Add the type to the `oneOf` array in `config.schema.json`.
6. Add any device-specific config parameters to the schema with appropriate `condition` functions.
7. Document the device in `docs/index.md` (supported devices table).

## Important Files to Keep In Sync

When making changes, ensure these files stay consistent:

| Change           | Files to update                                                                                                                                                                  |
| ---------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| New device type  | `src/accessories/{Name}.accessory.ts`, `src/accessories/index.ts` (re-export), `src/index.ts` (CLASS_DEF), `config.schema.json` (oneOf + params), `docs/index.md` (device table) |
| Rename platform  | `src/index.ts` (PLATFORM_NAME), `config.schema.json` (pluginAlias)                                                                                                               |
| New config param | `config.schema.json` (schema + condition), accessory class (read from context)                                                                                                   |
| New dependency   | `package.json`, verify `npm run build` still passes                                                                                                                              |

## Release and Documentation

- **CHANGELOG.md** is **auto-generated** by semantic-release in CI during version bumps. Do NOT manually edit it.
- **Commit messages** follow Conventional Commits format (`feat:`, `fix:`, `docs:`, etc.) to trigger automatic releases and changelog generation.
- **Version bumps** happen automatically via semantic-release when commits are pushed to master with the proper format.
- **Documentation** is built from `docs/` and deployed to GitHub Pages automatically on release.

## Things to Avoid

- Do not add `"type": "module"` to `package.json` — the project is CommonJS and Homebridge expects `require()`.
- Do not import from `homebridge` at the top level in accessory files — HAP types are injected at runtime via the registration function.
- Do not expose circuit breaker switch control to HomeKit — `CircuitBreakerMonitorAccessory` is intentionally read-only for safety.
- Do not create singleton `TuyaAccessory` instances — each device gets its own connection.
